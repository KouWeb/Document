# クリーンコードとは

-   読みやすい
-   機能追加が容易
-   バグ混入がしづらい

-   コメントに頼りすぎない。
    -   コメントが正しいとは限らない。
    -   メンテナンスされているとは限らない。
    -   コミットコメントで事足りることが多い。

-   影響調査にかけるコストを軽減できる。
-   コードレビューしやすくなる。

-   命名規則はクリーンコードの基礎。
-   何の値が格納される変数かがわかる変数名にする。

-   真偽値はisをつけた変数名にする。
-   定数はすべて大文字。

-   一般的・業界標準的な略語はあり。
    -   a11y → accessibility
    -   i18n → internationalization

-   if文は最終手段という認識を持つ。
-   条件分岐をあちこちで書かない。
-   よく使われる条件（適合数が多いもの）を上の方に記述する。

-   SOLID原則
    -   単一責任の原則
    -   オープン・クローズドの原則
    -   リスコフの置換原則
    -   インターフェース分離の原則
    -   依存関係逆転の原則

-   デザインパターンについて学ぶ。
