# テンプレート制作のルール

-   テンプレートの役割
    -   コーダーがデザインやレイアウトを意識しなくてもページ作成ができる
    -   誰が作業しても品質を一定に保つことができる
    -   コーディングに関するコミュニケーションコストを削減できる
-   開発環境
    -   ns-starter
-   基本的なコーディングルール
    -   インデントはタブのサイズ:4 を使用する（半角スペースは使用しない）
-   HTML コーディング（セマンティックなマークアップの方に集約？）
    -   ページ内リンクは id を使う
-   CSS 設計
    -   基本的な考え方（予測できる・再利用できる・保守できる・拡張できる）
    -   BEM（MindBEMding）
    -   FLOCSS
        -   SCSS ファイルの分け方
    -   命名規則・命名のコツ
        -   ケバブケース（ハイフン区切り）を使用する
        -   モジュールの特徴や役割が予測できる命名をする
        -   略語はできるだけ避ける
            -   img（image）や nav（navigation）など、HTML タグ名などにもあるような一般的な略称は許容する
        -   色や形など見た目を表現する命名は避ける
        -   CSS 用と JS フック用の class を分ける
-   CSS/SCSS コーディングルール
    -   リセット CSS を SCSS に含めない
    -   id セレクターを使用しない
    -   js フックにスタイルをつけない
    -   コメントアウトの使い分け
        -   モジュール名やコード内のセクショニング目的などの、コンパイル後も残しておきたいコメントは CSS コメント`/* */`を使う
        -   無効化したいコードやメモとして残しておきたいコードなど、コンパイル後は不要になるコードには SASS コメント`//`を使う（不要になったら削除してコードを綺麗にすること）
    -   モジュール毎にコメントをつける（`/* モジュール名 */`）
    -   ネストの書き方
    -   メディアクエリの使い方
    -   マージンの方向を統一する
    -   固定値をできるだけ避ける
    -   ホバーリアクションを必ずつける
    -   レスポンシブ対応
        -   メディアクエリをパーツ毎に記述する
        -   フォントサイズ調整
        -   グロナビなどで PC、SP で構造が変わる場合はソースごと分ける
-   JavaScript コーディングルール
    -   厳格モード（use strict）
        -   一部の非推奨処理などのミスや警告をエラーとして明示してくれる
        -   実行速度を高速化できる可能性がある
    -   スコープ
        -   処理のまとまり毎に`{ }`で囲むことでスコープを絞り、変数の汚染を防ぐ
    -   セレクター
        -   要素の取得はできる限り`getElementById`を使う
    -   ループ処理
        -   for ループと forEach ループの違い・使い分け
-   表示速度最適化（移管）
