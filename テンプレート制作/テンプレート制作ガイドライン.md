# テンプレート制作ガイドライン

## ガイドラインについて

このガイドラインは、Web 制作に必要な基本ルールをまとめたものです。

複数人が作業に関わる制作現場において、品質を一定にし、拡張や保守・運用がしやすい（誰がみてもわかりやすい）コードを書くことを目的としています。

案件特有のルールがある場合はそれに従い、必要に応じてルールの追加・変更してください。

ガイドラインの内容は定期的に見直し、アップデートしてください。

## テンプレートの役割

サイト制作におけるテンプレート（ページの雛形）は、以下の役割を持ちます。

-   コーダーがデザインやレイアウトを意識しなくても一貫性のあるページ作成ができる
-   ページ作成時に誰が作業しても品質を一定に保つことができる
-   コーディングに関するコミュニケーションコストを削減できる

テンプレート作成・修正時、コーダーによるページ作成時のいずれにしても、作業者によってコードの書き方がバラバラといった状態にならないように意識してください。

## コーディング作業を始める前に

作業に入る前に、以下の内容を確認して、テンプレート制作者間で意識合わせしてください。

### 要件・仕様関連

-   プロジェクト全体及びフェーズ毎のスケジュール
-   アクセシビリティ達成レベル
-   公開環境（イントラ or 外部？静的 or CMS？）
-   レスポンシブ対応要否
-   JavaScript による実装が必要な UI の有無（例：カルーセル、モーダルウィンドウ、アコーディオン、タブなど）
    -   ある場合は、プラグインやフレームワークを導入できるかの確認、導入できるのであればどれを使うかの選定
-   お客さん側で運用・保守をするかどうか

### 【移行済】開発環境関連

-   基本は ns-starter を使用
    -   Node モジュールを追加する場合は使い方を含めて共有
-   SVN のルール
    -   ページ毎、モジュール作成毎など、作業のキリがいいところでこまめにコミットしてください。
    -   関連性のない複数のページやファイルをまとめてコミットするのはできるだけ避けてください。
    -   ページやモジュール名など、作業範囲がわかるコミットメッセージを記載してください。
        -   例：グロナビ修正、テーブルコンポーネント修正　など
    -   できる限り、少なくとも 1 日に 1 度はコミットしてリポジトリを最新化してください（大きく表示崩れする、エラーで動かなくなるといった作業状況のものは保留）。

### デザイン関連

デザインデータがある場合は、以下のような内容を確認してください。

各種サイズ設定やメインで使用する色など全体で共通化できる設定は、必要に応じて変数化しテンプレートのベース設定に組み込んでください。

必要に応じてデザイン担当者と相談し、意識合わせをしてください。

デザインに関する全てのことをデザインパートに確認したり、デザインデータを準備してもらったりしてから作業をすると、デザインパート側の負担が大きくなりすぎたり進行が滞ったりします。

軽微な部分（ホバー時のリアクションやレスポンシブ時のレイアウト変化など）については、場合によってはコーダー側で仮組みしておいて、デザインパート側のフィードバックをもらって調整するような流れにしてください。

#### 事前に確認できると良い内容例

-   基本的なタイポグラフィ
    -   各種見出しのフォントサイズ、行間、上下余白など（PC/TB/SP）
    -   通常テキストのフォントサイズ、行間、上下余白など（PC/TB/SP）
-   全体の左右余白（PC/TB/SP）
-   セクション毎（見出し毎）の余白
-   主要な色（メインカラー/サブカラー/etc...）
-   インタラクティブコンテンツの動き
    -   リンクやボタンのホバー時
    -   ハンバーガーメニューやドロップダウンメニューの動き
-   TB/SP 対応時の振る舞い
    -   レイアウトが変わったときの見た目
    -   文字サイズや余白の変化
-   ヘッダーやフッターなど共通レイアウト化できる部分の把握
-   コンポーネントとして作成が必要な UI の把握
    -   可能であればデザインのキャプチャを含めリストアップ、分類しておくとベスト

## マークアップ

CSS によるデザインやレイアウトについて考える前に、まずはセマンティックタグを使って正しいマークアップをしてください。

適切な文書構造でコンテンツをマークアップできてから、スタイル付与のための div や class 属性を付け足していくことを心がけてください。

## CSS 設計

### 基本的な考え方

#### 予測できる

「スタイルの振る舞いを予測できる」、「スタイルの影響範囲を予測できる」ことを意識して設計してください。
class 名を見てもどこで定義され、どこで使用され、どのように振る舞っているかが予測できなかったり、新しいスタイルの追加や既存スタイルの変更を行った際に意図しない範囲に影響が出てしまう、といったことがないようにしてください。

#### 再利用できる

既存のパーツを別の箇所で使用したいとなったとき、コードの書き直しや上書きが必要ないようにしてください（BEM の Mix による調整は許容）。
子孫セレクターなどを使用しないようにし、モジュール間に依存関係がないようにしてください。

#### 保守できる

モジュールや機能を追加・更新する際に、既存のモジュールのリファクタリングが必要ないようにしてください。
あるモジュールを追加・変更した際に、他のモジュールが壊れてしまうことがないようにしてください。

#### 拡張できる

複数人で作業する場合でも、問題なくソースコードを管理できる状態にしてください（命名規則などのルールを守ってください）。
後から作業に入った人でも、どのように管理されているかが理解でき、低い学習コストで作業に入れるようにしてください。

### 設計手法

FLOCSS のルールに従って SCSS ファイルや各モジュールを管理してください。

命名規則は FLOCSS のプレフィックス＋ MindBEMding に従ってください。

### レスポンシブ対応について（メモ）

モジュール毎にレスポンシブ対応のコードを書いてください。

デザイン、レイアウトで迷うことがあればデザイン担当者に確認してください。

レスポンシブ用のデザインが用意されてない場合でも、複雑なレイアウトやデザインでない場合はコーダー側でいったん作成してデザイナーにレビューしてもらった方がスムーズ。
