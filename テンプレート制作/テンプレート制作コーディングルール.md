# テンプレート制作コーディングルール

## 基本的なコーディングルール

### インデント

インデントはタブ（サイズ：4）を使用してください。半角スペースは使用しないでください。

### パス

サイト内リソースへのアクセスは、ルートパスを使用してください（CSS や JS ファイルの読み込み、リンク、画像など）。

イントラサイトなど、ドメイン以下の深いディレクトリにサイトルートがあり、ルートパスが使用できない場合は、JS で相対パスに書き換える処理をしてください。

### 特殊文字

UTF-8 でエンコードされていて、文字化けがおきない場合は、記号を文字参照に変換する必要はありません。

変換したほうがいい場合のある記号例

-   <（半角）：HTML タグと誤認される場合があるため。
-   &（半角）：文字参照と誤認される場合があるため。

### 全角・半角

基本的に、英数字は半角に統一してください。  
記号やスペースは文脈によって全角・半角を使い分けてください。  
固有名詞や原稿で意図的に使い分けられている場合は、それらに合わせてください。

## HTML 関連

### セマンティクス

可能な限りセマンティックタグを使用してください。

文書構造的に意味を持たないマークアップをしないでください（`<div>`の中に直接テキストなど）。

### 属性の記述順

属性は以下の順で記述してください。

class > id > その他

### スタイル付与

スタイルを付与する場合は class 属性を使用してください。  
原則、スタイルの付与に id 属性や style 属性は使用しないでください。

### アンカー

アンカーは id 属性を使用してください。  
`<a name="">`は使用しないでください（廃止済）。

### 画像関連

#### alt 属性

alt 属性は省略しないでください。  
文脈上意味のある画像には、必ず alt 属性及び画像を説明するテキストを記述してください。  
文脈上意味のない装飾的な画像には、必ず空の alt 属性を記述してください。

#### レスポンシブ対応

デバイスサイズによって異なる画像を読み込ませる場合は、`<picture>`及び`<source>`要素を使用してください。

#### width 属性、height 属性

`<img>`要素、`<picture>`要素には、必ず width 属性、height 属性を記述してください。

## CSS / SCSS 設計関連

### 設計手法

FLOCSS のルールに従ってファイル、モジュールを管理してください。

命名規則は BEM（MindBEMding）のルールに従ってください。

### エンコード

エンコードは UTF-8 を使用してください。  
必ずファイルの 1 行目に`@charset "UTF-8";`を記述してください。

### SCSS 記法

SASS を使用する場合、SCSS 記法を使用してください。

### リセット CSS をコンパイルしない

リセット CSS を SASS コンパイルに含めないでください。  
ファイルを分けて HTML に読み込ませてください。

### コメントアウトの使い分け

CSS コメントと SASS コメントを使い分けてください。

コンパイル後も残しておきたいコメントには CSS コメント`/* */`を使用してください。

-   コード内のセクショニング目的の罫線
-   モジュール名

コンパイル後は残しておく必要のないコメントには SASS コメント`//`を使用してください。

-   無効化したいコード
-   メモとして開発環境上にだけ残しておきたいコード

### モジュール名をコメントで記載

モジュール毎に、なんのモジュールなのかがわかる名前を CSS コメントで記述してください。

```scss
/* 送信ボタン */
.c-button-submit {
	...;
}

/* キャンセルボタン */
.c-button-cancel {
	...;
}
```

### メディアクエリはモジュール毎に記述

メディアクエリは、モジュール毎にスタイルブロック内に記述してください。

```scss
/* カード */
.c-card {
	...;
	// ↓メディアクエリのミックスイン
	@include g.media-df(sp) {
		...;
	}
}
```

### モジュールのコードをまとめる

モジュール毎に、そのモジュールに関連する全てのスタイルが、ブロック内にネストされまとまるようにしてください。

```scss
// OK
/* カード */
.c-card {
	...;
	@include g.media-df(sp) {
		...;
	}
	&__heading {
		...;
	}
	&__description {
		...;
	}
}

// NG
/* カード */
.c-card {
	...;
}
.c-card__heading {
	...;
}
.c-card__description {
	...;
}
@include g.media-df(sp) {
	.c-card {
		...;
	}
	.c-card__heading {
		...;
	}
	.c-card__description {
		...;
	}
}
```

## セレクター

### class セレクターを使用する

スタイルの付与は必ず class セレクターを使用してください。

### id セレクターを使用しない

スタイルの付与に id セレクターを使用しないでください。

### タイプセレクターを使用しない

スタイルの付与にタイプセレクター（タグ名セレクター）を使用しないでください。  
スタイルを付与する場合は必ず class 属性を使用してください。

### 子孫セレクターを使用しない

原則、子孫セレクターなど、スタイルの適用範囲が他の要素に依存するセレクターを使用しないでください。

## スタイリング関連

### 共通プロパティ値の変数化

サイトのメインカラーなど、複数箇所で使用され共通化できる値は SASS 変数にしてください。

#### 例：global/variables/\_color.scss

```scss
$color-primary: #xxxxxx;
$color-secondary: #xxxxxx;
```

### 汎用パーツにマージンやポジション、固定のサイズを設定しない

サイト内で再利用される汎用パーツには、マージンや幅・高さなど、周囲の要素に影響を与える設定をしないでください。

サイトのどこで使用しても同じように振る舞うよう、最低限のスタイルを付与するように心がけてください。

### マージンの方向を下に統一

原則、セクション間やモジュール間の余白は、下方向のマージンに統一してください。

上下マージンが混在していると、マージンの相殺により、ファクタリング時やレスポンシブ用スタイル調整時に意図した制御がしづらくなる場合があります。

### インタラクティブコンテンツのリアクション

リンクやボタンのホバーリアクション、フォームのフォーカス時のスタイルなど、ユーザーが操作するコンテンツに対して、必ずリアクションのスタイルを付与してください。

### レスポンシブ対応

モジュール毎にレスポンシブ対応用のスタイルを設定してください。

## 命名規則

FLOCSS 及び BEM（MindBEMding）のルールに従ってください。

その他の細かなルールについては、以下を参照してください。

### 小文字の英単語を使用

ファイル名やモジュール名には小文字の英単語を使用してください。

```html
<!-- OK -->
<button class="c-button">ボタン</button>

<!-- NG -->
<button class="c-Button">ボタン</button>
```

### 単語の接続はケバブケース

ファイル名やモジュール名に複数の単語を使用する場合は、ケバブケース（ハイフンで接続）を使用してください。

```html
<!-- OK：ケバブケース -->
<button class="c-button-submit">ボタン</button>

<!-- NG：スネークケース -->
<button class="c-button_submit">ボタン</button>

<!-- NG：キャメルケース -->
<button class="c-buttonSubmit">ボタン</button>
```

### 略語を使用しない

略語の使用は避けてください。

img（image）や nav（navigation）など、HTML タグ名などにもあるような一般的な略称は許容しますが、プロジェクト内で表記揺れがおきないように注意してください。

```html
<!-- OK -->
<p class="u-text-right">右寄せテキスト</p>

<!-- NG -->
<p class="u-text-r">右寄せテキスト</p>
```

### SCSS ファイル名と関連付け

SASS 及び FLOCSS で SCSS ファイルを分けてを管理している場合、モジュール名の 1 単語目を SCSS ファイル名と一致させてください。

これによりどのファイルで管理されているモジュールなのかが明確になり、メンテナンス性が向上します。

#### component/\_list.scss

```scss
.c-list-module-name {
	...;
}
```

#### project/\_top.scss

```scss
.p-top-module-name {
	...;
}
```

### 分類＞性質・役割　の順で命名

モジュール名や変数名に複数単語使用する場合は、「それが何か」（カテゴリー）に続けて、「どんな役割か」「何に使われるか」などを説明するような命名にしてください。

1 単語目は抽象的な単語、2 単語目以降はより具体的な単語になるイメージです。

以下のように、色なら color、ボタンなら button と、同じ分類のモジュール名の 1 単語目が同じになり、コードの可読性が向上します。

```scss
// OK
$color-primary: #xxxxxx;
$color-secondary: #xxxxxx;

.c-button-submit {...}
.c-button-cancel {...}

// NG
$primary-color: #xxxxxx;
$secondary-color: #xxxxxx;

.c-submit-button {...}
.c-cancel-button {...}
```

### 機能・役割を示す命名

モジュール名は機能や役割を示す名前にしてください。  
具体的な見た目を示す名前は使用しないでください。

あとから色や形などのデザインが変更されたときに、モジュール名（class）と実際の見た目に乖離が発生しない命名を意識してください。

```scss
// OK
.c-text-error {...}
.c-button-submit {...}

// NG
.c-text-red {...}
.c-button-blue {...}
```

モディファイアは、見た目を示す命名を許容します。

```scss
// OK
.c-button--square {...}
.c-button--round {...}
```
